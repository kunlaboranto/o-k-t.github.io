문제점

설명/조치

패치 PHCO_33675
및 PHCO_34718에서의 성능 저하

HP-UX 11i v2 시스템에 PHCO_33675 패치나
최신 libpthread 패치 PHCO_34718을 설치한 경우 로드된 시스템의 응용
프로그램에서 공유 mutex를 더 오랫동안 보관하므로 성능이 저하될 수
있습니다. 공유 mutex를 두고 경합하는 스레드가 고속으로 회전하기
때문에 이 문제가 발생합니다.

수행할
작업

VxFS 3.5 복원

2006년 9월 릴리즈의 경우 OnlineJFS 4.1 제품이 HP-UX 11i v2 Mission Critical Operating Environment(MCOE)
및 Enterprise Operating Environment(EOE)와 함께 항상 설치됩니다.
그러나 미디어에서 MCOE 또는 EOE의 2006년 3월 또는 2006년 6월 릴리즈를
설치하는 경우에는 OnlineJFS 버전 3.5나 4.1 중에서 설치할 버전을 선택해야 합니다.
이러한 버전의 제품은 MCOE 또는 EOE 미디어에서 자동으로 설치되지
않습니다.미디어에서 HP-UX 11i v2 2006년 9월 버전의 MCOE 또는 EOE를
초기 설치하거나 업데이트하면 소프트웨어 종속성에 따라 OnlineJFS 4.1 및 VxFS 4.1(EnableVXFS 포함)가 자동으로 설치됩니다. 시스템을 VxFS 3.5 및 OnlineJFS 3.5로 복원하려면 OnlineJFS 4.1 및 Base-VXFS를 제거하고 시스템을 재부팅한 다음 코어 OS 미디어에서 OnlineJFS 3.5를 다시 설치해야 합니다. 자세한 내용은 IT 리소스
센터 웹 사이트(http://itrc.hp.com )를 참조하십시오.

    위에 나열된 제품을 제거합니다. 또는

  • 위에 나열된 제품을 제외한

    update-ux -s depot -f new_selections_file HPUX11i-OE_Type

위에서 설명한
작업은 PA-RISC 시스템에서만 수행해야 합니다. Itanium 시스템에서는
이 작업을 수행하지 마십시오.

Servicecontrol
Manager 2.5 또는 이전 버전이 설치된 상태에서 업데이트할 수 없음

HP Servicecontrol Manager 2.5 이전 버전이
HP-UX 11i v1 운영 체제에 설치되어 있는 경우 또는 HP Servicecontrol
Manager 2.5가 HP-UX 11i v1 운영 체제에 구성되어 있는 경우 HP-UX
11i v2로 시스템을 업데이트하기 전에 Servicecontrol Manager를 제거해야
합니다. 시스템에서 HP Servicecontrol Manager 2.5 또는 이전 버전을
제거하지 않는 경우 HP-UX 11i v2로의 업데이트 프로세스가 실패합니다.

HP-UX
11i v1에서 HP-UX 11i v2로 시스템을 업데이트하는 중이고 HP Servicecontrol
Manager를 한 번도 실행하지 않았으며 HP Systems Insight Manager를
사용할 계획이 없는 경우 운영 체제를 HP-UX 11i v2로 업데이트한 후에
HP Servicecontrol Manager 제품을 제거할 수 있습니다.

수행할 작업

다음 명령을
입력하여 HP Servicecontrol Manager 2.5 또는 이전 버전이 시스템에
설치되어 있는지 확인합니다.

swlist -l product ServControlMgr AgentConfig \
SysMgmtServer SysMgmtAgent

버전 번호가 A.02.05.xx 또는 이전인지 확인합니다.
버전 번호가 A.02.05.xx 또는 이전일 경우 HP-UX 11i v1 시스템에서 Servicecontrol
Manager를 제거해야 합니다.

또한 /var/opt/mx/data/.inititalization 파일이 있는지 확인합니다. 시스템에 이 파일이 있는 경우
HP-UX 11i v2로 업데이트하기 전에 HP-UX 11i v1 시스템에서 Servicecontrol

여기에서 ID는 제품 또는 번들 ID입니다.

rm -fr /opt/mx /etc/opt/mx 명령을 실행하여 이전 제품의 하위 디렉토리를 제거합니다.

HP Servicecontrol
Manager 3.0 및 HP Systems Insight Manager 4.x에서의 업데이트 문제

HP-UX 11i v1에서 HP-UX 11i v2로 업데이트하는
중이고 HP Systems Insight Manager(HP SIM) 4.x가 설치되어 있으며
이후에 HP SIM을 실행하지 않으려는 경우 HP SIM 제품을 제거할 수 있습니다.
HP SIM을 제거하려면 swremove -x enforce_dependencies=false T2414BA 명령을 실행합니다.

HP-UX 11i v1에서 HP-UX
11i v2로 업데이트하는 중이고 시스템에 HP Servicecontrol Manager
3.0이 구성되어 있거나 HP SIM이 설치되어 있으며 이후에 HP SIM을 계속
실행하려는 경우 다음 업데이트 시나리오 중 하나를 사용하여 HP SIM으로
업데이트해야 합니다.

HP Servicecontrol Manager 3.0이
시스템에 구성되어 있는 경우 데이터를 손실하지 않도록 하려면 HP-UX
11i v1에서 HP-UX 11i v2로 시스템을 업데이트하기 전에 Servicecontrol
Manager 3.0을 HP SIM 4.2로 업데이트해야 합니다. 자세한 내용은 http://docs.hp.com 에 있는 HP SIM Installation and User Guide를 참조하십시오.

HP
Servicecontrol Manager의 데이터를 보관하지 않으려는 경우 swremove ID 명령을 사용하여 업데이트를 시작하기 전에 Servicecontrol
Manager를 제거할 수 있습니다.

Servicecontrol Manager가
구성되어 있지 않거나 HP SIM 4.x가 설치되어 있고 HP SIM을 계속 사용하려는
경우 업그레드하려면 HP SIM 5.x를 선택해야 합니다. -x match_target=true 옵션을 사용하거나 대화식 모드를 사용하여 HPSIM-HP-UX를
선택함으로써 이 작업을 수행합니다. 명령줄에서 HPSIM-HP-UX를 지정할
수도 있습니다. HP SIM 5.x 설치 프로그램은 구성되어 있는 HP SIM 4.x
및 구성되어 있지 않은 HP SIM 4.x를 모두 업그레이드합니다. HP Servicecontrol
Manager 또는 HP SIM을 제거하려는 경우 rm -fr /opt/mx /etc/opt/mx 명령을 실행하여 제품에서 사용된 하위 디렉토리를 제거해야
합니다. 이러한 하위 디렉토리를 제거하지 않은 경우 HP SIM 5.x를 설치할
때 mxinitconfig -a 명령을 실행하려 하면 데이터베이스 오류가 발생할 수 있습니다.

OE 미디어 설치
도구와 네트워크 저장소가 일치해야 함

OE 미디어의 Ignite-UX
설치 도구는 네트워크 저장소 사용을 지원합니다. 그러나 설치 도구의
선택 사항이 제대로 동작하려면 네트워크 저장소가 미디어 저장소와
일치해야 합니다.

또한 Ignite-UX 설명서에서는 설치할
때마다 OE 미디어를 사용할 필요가 없도록 네트워크 저장소를 설치하여
사용하는 것에 대해 설명하고 있습니다. Ignite-UX 제품에 대한 자세한
내용은 다음 웹 사이트를 참조하십시오.

http://docs.hp.com/en/IUX/

루트("/")와 부팅("/stand") 디렉토리를 결합하는 것은 좋지 않음

볼륨 관리를 사용하는 사용자 정의 파일
시스템 구성을 만들 때 루트("/")와 부팅("/stand") 디렉토리를 같은 파티션에 두지 않는 것이 좋습니다.
볼륨 관리가 활성 상태인 경우 기본 구성은 루트와 부팅 디렉토리를
서로 다른 파티션에 둡니다.

수행할
작업

HP-UX를 설치할 때 기본 파일 시스템 구성을
사용하는 것이 좋습니다. 볼륨 관리를 사용하는 사용자 정의 파일 시스템
구성을 만들려는 경우에는 루트와 부팅 디렉토리가 서로 다른 파티션에
있는지 확인합니다.

OnlineDiag 번들
및 SysFaultMgmt 번들로 로드해야 함

OnlineDiag 번들 또는 SysFaultMgmt 번들을 제거하거나 설치하지 않으면 이러한 번들에 종속된
일부 제품이 제대로 작동하지 않습니다.

최신 업데이트를
구하려면 다음 진단 설명서를 참조하십시오.

http://docs.hp.com/en/diag.html

reboot 명령 대신
shutdown 명령 사용

드문 경우지만 종료 중에 치명적인 하드웨어
또는 운영 체제 문제가 발생하면 reboot 명령이 실행되지 않을 수 있습니다.이전 버전의 HP-UX에서는
10분 내에 시스템이 완전히 종료되지 않으면 시간이 초과되고 시스템이
응답하지 않을 수 있습니다. 지금은 시스템이 종료 프로세스가 완료될
때까지 무한정 기다립니다.

수행할
작업

reboot 명령은 사용하지 않는 것이 좋습니다. 대신 shutdown 명령을 사용합니다.

HPUX-HIDS로 인해
자동 재부팅이 수행됨

2004년 9월 릴리즈에서 HIDS 제품 번들이
재구성되었기 때문에 HPUX-HIDS 버전 2.x에서 HPUX-HIDS의 새로운 버전으로
설치 또는 업데이트하면 시스템이 자동으로 재부팅됩니다. 이전 제품
릴리즈에서는 업데이트 후에도 고립된 파일이 시스템에 계속 남아 있었으며 이러한
문제를 해결하기 위해 제품 재구성이 구현되었습니다. HPUX-HIDS 버전
3.0에서 HPUX-HIDS의 새로운 버전으로 업데이트하는 경우에는 시스템
재부팅이 필요 없습니다.

VxFS 및 18GB
디스크에서 Whole Disk 설치가 지원되지 않음

VxFS 및 18GB의 디스크만 포함하는 Whole
Disk 설치 구성으로는 HP-UX를 설치할 수 없습니다.18GB 디스크에서
VxFS의 Whole Disk 설치를 시도하면 설치 프로세스 중 여러 곳에서 설치가
정지할 수 있습니다.

수행할 작업

18GB
디스크에서는 Whole Disk 옵션을 사용하여 VxFS를 설치하는 것을 피해야
합니다. 대신 LVM 또는 VxVM을 사용하여 디스크 파티션을 지정하는 설치
구성을 권장합니다.

last_install
구성이 이미 존재함 경고

HP-UX 11i v2로 업데이트하는 동안 다음
경고가 swagent.log에 나타날 수 있습니다.

WARNING: The configuration named 'last_install'
already exists.
NOTE: Continuing since -f flag was specified.
* Please wait: flushing kernel registry database

'last_install'.
* The title of the configuration 'last_install'

업데이트에 이전 버전의 구성이 이미 존재하기 때문에 경고가 발생합니다.

이

MirrorDisk/UX(B2491BA
또는 B5403BA) 설치 또는 제거

합니다. LVM 명령 패치는 PHCO_31709 또는 대체된 패치일 수 있습니다.

LVM
명령 패치를 다시 설치하지 않으면 다음 두 개의 문제가 발생할 수 있습니다.

시스템에
존재하는 LVM 명령 패치나 최신 패치를 다시 설치해야 합니다. 시스템에서
MirrorDisk/UX 제품을 제거하면 -x reinstall=true 옵션으로 LVM 명령 패치를 설치해야 합니다.

LVM
명령 패치 PHCO_31709 또는 대체 패치는 HP-UX 11i v2 OE 미디어 2004년
9월 이상에 있는 BUNDLE11i나 http://itrc.hp.com 의 IT Resource Center 웹 사이트에서 확인할
수 있습니다.

대형 메모리 시스템에서
update()가 응답하지 않음

재부팅하는 동안 update() 함수 호출 실패로 인해 reboot r 프로세스를 수행하는 중에 대형 메모리 시스템이 응답하지
않을 수 있습니다. 이 오류는 update()가 완료되기 전에 타이머 값(UpdateHangTimeout)이
만료되어 발생합니다.

이 문제는 128GB의 메모리가 설치된
rx8620 32-웨이 서버에서 발생하는 것으로 알려져 있습니다.

이
문제를 완화하려면 다음(굵게 표시된 것은 사용자 입력)과 같은 adb 명령을 사용하여 UpdateHangTimeout 타이머
값을 20으로 늘리십시오.

adb -w /stand/vmunix /dev/kmem
abd> UpdateHangTimeout /W 0x1d4c0
UpdateHangTimeout:0x120000=0x1d4c0
abd> UpdateHangTimeout/D
UpdateHangTimeout:
UpdateHangTimeout:        120000

참고: 부하가 높은 경우 재부팅하는 동안 시스템이
최소 20분간 계속 응답하지 않을 수 있습니다.
