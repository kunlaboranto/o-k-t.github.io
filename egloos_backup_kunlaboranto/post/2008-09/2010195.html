1.1 데이터 블록 크기

Oracle9i는 같은 DB 내에서 다른 db block size를 사용할 수 있다. Tablespace 생성시 BLOCKSIZE 를지정함으로써 사용하고자 하는 db block size 를 정의할 수 있다. BLOCKSIZE에 지정된 block size는db_nk_cache_size파라미터에 설정된 값이어야 하며 temporary tablespace 에 대해서는BLOCKSIZE를 지정할 수 없다

-.large block size가 유용한 경우

  a.tablescan이 빈번하거나 읽기 위주일 때 사용하면 어플리케이션이 디스크나 다른 cache로부터 데이터를 읽는 양이 줄어들기 때문에결과적으로 Disk로부터의 block read를 위한 lock overhead가 줄어들게 된다

  b.Dataload나 queue 테이블에 대용량의 데이터가 insert되거나 write concurrency가 없는 경우 사용하면 새로운Space가 할당되기 전 한 block에 더 많은 데이터를 넣을 수 있어 유리하며 특히 DW 어플리케이션에서 유용하다

-.Small block size가 유용한 경우

  a.같은 block 내의 data를 자주 access하는 어플리케이션에서 사용하면 disk로부터 더 많은 block을 읽어야 하지만 경합의 가능성은 줄일 수 있다

1.3 Undo Tablespace

-.오라클 9i 부터는 undo 정보를 저장하는 방법으로 기존의 manual 방식(rollback segment사용)과 오라클이 자동으로 관리하는 automatic 방식(undo tablespace 사용) 중 하나를 선택할 수 있다

-.Automatic방식은 자체적으로 롤백(Rollback) 세그먼트를 관리하기 때문에 더 이상 DBA가 신중히 롤백 세그먼트의 수 및 크기를계획하고 조정하지 않아도 되며, 트랜잭션을 어느 특정 롤백 세그먼트에 전략적으로 할당해야 하는지에 대해 고민할 필요가 없다.또한 undo정보가 유지되는 시간(UNDO_RETENTION)을 설정함으로써 undo 블록에 대한 충돌을 최소화할 수 있다는장점이 있다

-.Automatic undo management 방식을 사용하는 경우 제약사항은 아래와 같다

  a.Undo tablespace에 임의로 object를 생성할 수 없다

  b.Tablespace생성시 datafile절만 지정하며 자동으로 LMT방식을 사용한다

  c.RAC 내의 모든 인스턴스는 동일한 undo management 방식으로 운영되어야 하며 각 인스턴스는 개별 undo tablespace를 사용한다
