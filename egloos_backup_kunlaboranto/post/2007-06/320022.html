They are a statistic that can be interesting but not too much of a problem if you ignore them. if memory serves me correctly. The way I understand this is that the system thought it would be really nice if it could grab a 16K size page but sadly there weren't any free so it had to grab a 64K one and so on. The idea being that if most of your requests need a 256K page to fit in then it is a good idea to have more 256K pagaes available. However, I don't even believe it is possible to tune it so I don't think you can do much about this anyway. Could be wrong though.

vmstat -s basically just tells you about ALL paging activities since that last boot OR the last time vmstat -z was run.
And the failures are simply times when a process asked for those size pages & was unable to get them for any number of reasons such as amount free or contigous or such.

It appears to me from your output, that you have one whopper of a memory hungry app that is wanting a 64M page and your system is putting it on a 4K diet. You may need more memory or a reboot to unfragment your memory.

Run vmstat -z to clear the stats & check it again the next day & so where those stats stand. If it is still wants 64M pages & you *never* have that much free then you *may* need more RAM.
If you take a reboot when able and it appears you have plenty of free memory then check the stats again the next day & if the 64M is low BUT keeps growing faster & faster, that's a good sign that something is leaking and/or fragmenting memory.

There is a new feature called, Variable Size Pages, that was added in 11.00. This feature gives the application the ability to request a given page size for use when it executes. In the past the page size was fixed to be 4KB, and every process was stuck with that default page size.
See the section titled "Variable Size Pages" in the 11.00RelNotes file in /usr/share/doc for more info.

There are 3 new kernel tunable parameters that control the behavior of Variable Size Pages:

vps_ceiling
vps_chatr_ceiling
vps_pagesize (the default page size is 4KB)

Also, see the chatr man page for details on how to set the data page size and the instruction page size for individual programs. The options are "+pd" and "+pi".

If you know that you application requires larges contiguous chunks of memory, you might consider increasing the "+pd" value for that process. This may or may not increase system performance since the OS will try to determine the optimum page size for a given process even if the chatr value is not set.
